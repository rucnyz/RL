# Requirements for the harbor_agent venv used by recipe B
# (`grpo-qwen3.5-9b-1n8g-harbor-e2b.yaml`).
#
# Why this file lives in research/pgc_swe/ instead of the upstream
# 3rdparty/Gym-workspace/Gym/responses_api_agents/harbor_agent/:
#   We pin a different harbor commit than upstream and need an openai
#   override to make it resolve. Keeping our pins inside research/pgc_swe/
#   avoids dirtying the Gym submodule and means we can PR / share this
#   project without forcing reviewers to also accept submodule deltas.
#
# The launcher (`run_harbor_e2b.sh`) does:
#   uv pip install -r research/pgc_swe/configs/harbor_agent_requirements.txt \
#       --override research/pgc_swe/configs/harbor_agent_overrides.txt \
#       "ray==${NEMORL_RAY_VERSION}"
# (the ray version is pinned at install-time to whatever the main NeMo RL
# venv resolved to, so the harbor_agent Ray actor and the trainer agree on
# Ray protocol version)

# nemo-gym in editable mode from the Gym submodule (provides the agent
# scaffolding + responses-api adapter). nemo-gym's pyproject pins
# `openai<=2.7.2`; the override file forces a newer openai so that latest
# harbor's litellm requirement resolves.
#
# `../..` is resolved relative to the current working directory, so the
# launcher (`run_harbor_e2b.sh`) cd's into
# `3rdparty/Gym-workspace/Gym/responses_api_agents/harbor_agent/` before
# running `uv pip install -r <this_file>`. From there, `../..` points at
# the Gym checkout — the same mechanism the upstream
# `responses_api_agents/harbor_agent/requirements.txt` uses.
-e ../..[dev]

# Pin to harbor 9dddd79 (2025-08-04). NOT latest — see below.
#
# Why not latest (5e21c0b, 2026-04-30):
#   harbor 0.6.x landed 466 commits over 9 months that touched mostly
#   features/adapters we don't use (TensorLake/Modal/Singularity/GKE
#   environments, codex/gemini-cli/opencode agents, ABC-Bench/AA-LCR
#   adapters, ATIF trace export, Windows tasks). The two architectural
#   refactors that *do* affect us are breaking changes nemo-gym 0.3.0rc0
#   has not adapted to:
#     * PR #1285: collapsed LocalDatasetConfig + RegistryDatasetConfig
#       into a single DatasetConfig. nemo-gym imports the old names
#       from the concrete module path → ImportError on every /run.
#     * PR #1235: removed OrchestratorConfig (replaced by inline
#       JobConfig fields + TrialQueue abstraction). nemo-gym still
#       constructs an OrchestratorConfig instance → ImportError.
#   Plus a separate `RuntimeError: Set changed size during iteration`
#   we hit during E2B sandbox bookkeeping. Living with the breakage by
#   maintaining a growing pile of compat shims wasn't worth it.
#
# What we lose vs latest:
#   For our specific stack (Terminus-2 agent + E2B environment +
#   nemotron scientific_computing dataset), basically nothing. The only
#   feature we wanted from a newer harbor was PR #1397's
#   `Template(file_context_path=...)` fix (so E2B template builds use
#   the per-task `environment/` dir instead of harbor's install dir).
#   That's a one-symbol monkey-patch, applied at venv-build time by
#   research/pgc_swe/scripts/patch_harbor_for_nemo_gym.py.
#
# NB: 9dddd79 keeps `e2b>=2.4.2` as a HARD dependency in `[project]`, so
# the bare `harbor` install is enough — no `[e2b]` extra needed (the extras
# split happened later in 0.6.x). Trying `harbor[e2b]` against 9dddd79
# fails resolution with "no such extra".
harbor @ git+https://github.com/laude-institute/harbor.git@9dddd797b57ab8a0f9d6352a20fce73abbb29573
